
## 十二、RISC-V 内存布局（Memory Layout）
RISC-V 32位架构的虚拟地址空间从低地址到高地址，分为5个核心段，各段用途与特性如下：

| 内存段 | 英文全称 | 核心用途 | 关键特性 |
|--------|----------|----------|----------|
| 保留段 | Reserved | 系统保留，用于异常处理、非法地址捕获 | 低地址区域，用户程序不可访问 |
| 代码段 | Text Segment | 存放程序的机器指令（二进制代码） | 只读、可执行，固定大小，由PC寄存器指向，起始地址通常为`0x00400000` |
| 静态数据段 | Static Data Segment | 存放全局变量、静态变量、常量数组与字符串 | 生命周期贯穿程序全程，`$gp`(全局指针)初始化为该段基地址`0x10008000`，支持±偏移快速访问 |
| 动态数据段（堆） | Dynamic Data (Heap) | 存放动态分配的内存 | 由`malloc()/free()`（C）或`new/delete`（Java）管理，从低地址向高地址增长，常见问题包括内存泄漏、悬空指针（Dangling pointer） |
| 栈段 | Stack | 存放函数的局部变量、过程调用的栈帧、返回地址与寄存器现场 | 自动分配与释放，栈指针`$sp(x2)`始终指向栈顶，从高地址`0x7fffffff`向低地址增长；<br>**push操作**：`sp`减去对应字节数；<br>**pop操作**：`sp`加上对应字节数 |

## 十三、存储程序计算机（Stored Program Computers）
### 13.1 核心定义
存储程序计算机，又称**冯·诺依曼架构（Von Neumann Architecture）**，以数学家与计算机先驱约翰·冯·诺依曼命名，核心思想是：**指令和数据以二进制形式无差别地存储在同一个内存中**。
### 13.2 核心特性
1. **二进制统一编码**：指令和数据都用二进制表示，硬件无需区分二者，仅通过PC寄存器识别指令，通过Load/Store指令识别数据
2. **程序可操作程序**：编译器、链接器等程序可以将其他程序（如C源码）作为输入数据处理，生成新的可执行程序
3. **二进制兼容性**：标准化的ISA使得编译后的程序可以在所有兼容该ISA的计算机上运行，无需重新编译
4. **通用计算能力**：通过修改内存中的程序，同一台硬件可以执行完全不同的任务，无需重新设计电路

## 十四、C程序的完整翻译与执行流程
一个C语言程序从源码到在内存中执行，需要经过**编译器(Compiler)、汇编器(Assembler)、链接器(Linker)、加载器(Loader)** 四个核心阶段，全流程如下：
```
C程序(.c) → 编译器 → 汇编程序(.s/.asm) → 汇编器 → 目标模块(.o/.obj) → 链接器 → 可执行文件(.out/.exe) → 加载器 → 内存执行
```

### 14.1 编译器（Compiler）
- **输入**：C语言源文件(.c)
- **输出**：RISC-V汇编语言文件(.s/.asm)
- **核心工作**：将高级语言代码翻译为符合ISA规范的汇编代码，完成语法分析、语义分析、优化、寄存器分配等工作

### 14.2 汇编器（Assembler）
- **输入**：汇编语言文件(.s/.asm)
- **输出**：可重定位目标模块(.o/.obj)
- **核心工作**：将汇编指令翻译为二进制机器码，同时生成辅助信息用于后续链接；release模式下会移除debug信息与label标签，生成的汇编代码无符号信息。

#### 目标模块（Object Module）的结构
| 组成部分 | 英文名称 | 核心用途 |
|----------|----------|----------|
| 头部 | Header | 描述目标模块的内容、各段的大小与位置 |
| 代码段 | Text Segment | 翻译后的二进制机器指令 |
| 静态数据段 | Static Data Segment | 程序生命周期内常驻的数据 |
| 重定位信息 | Relocation Info | 记录依赖于程序加载绝对地址的内容，用于链接器修正地址 |
| 符号表 | Symbol Table | 记录全局定义的符号、外部引用的符号（如函数名、全局变量名） |
| 调试信息 | Debug Info | 关联机器码与源代码的信息，用于调试，release模式下通常会被移除 |

### 14.3 链接器（Linker）
- **输入**：多个目标模块、静态/动态库文件(.a/.so/.lib/.dll)
- **输出**：可执行文件(.out/.exe)
- **核心工作**：
  1. 合并多个目标模块的代码段、数据段
  2. 解析符号标签，确定所有符号的绝对内存地址
  3. 修正位置依赖的地址与外部引用，完成重定位

#### 静态链接 vs 动态链接
| 特性 | 静态链接（Static Linking） | 动态链接（Dynamic Linking） |
|------|------------------------------|------------------------------|
| 库文件处理 | 将所有依赖的库代码完整打包到可执行文件中 | 不打包库代码，仅记录依赖的动态库(.so/.dll)，运行时由操作系统加载 |
| 文件体积 | 大，通常几MB到几十MB | 小，通常仅KB级 |
| 可移植性 | 好，单文件即可运行，无依赖 | 一般，需目标系统包含对应的动态库 |
| 内存占用 | 高，每个程序都有一份库的副本 | 低，多个程序共享同一份动态库内存 |
| 更新维护 | 麻烦，库更新需重新编译整个程序 | 方便，库更新无需修改程序本身 |

### 14.4 加载器（Loader）
- **输入**：可执行文件
- **输出**：内存中可运行的程序进程
- **核心工作步骤**：
  1. 读取可执行文件的头部，确定代码段、数据段的大小
  2. 为程序创建虚拟地址空间
  3. 将代码段和初始化数据复制到内存中，或设置页表项实现按需缺页加载
  4. 在栈上设置程序启动参数（argc、argv）
  5. 初始化寄存器（包括`$sp`栈指针、`$fp`帧指针、`$gp`全局指针）
  6. 跳转到启动例程，将参数传递给`main`函数；`main`返回后执行exit系统调用

## 十五、汇编器伪指令（Assembler Pseudoinstructions）
### 15.1 核心定义
伪指令（Pseudoinstructions）是汇编器提供的“虚拟指令”，**不对应真实的硬件机器指令**，是汇编器为了简化编程而提供的语法糖，汇编时会被翻译为一条或多条真实的机器指令。
### 15.2 常见伪指令与对应真实指令
| 伪指令 | 功能 | 对应的真实机器指令 |
|--------|------|---------------------|
| `mv rd, rs` | 寄存器间数据移动，`rd = rs` | `add rd, rs, x0` |
| `li rd, imm` | 加载立即数到寄存器，`rd = imm` | 12位立即数：`addi rd, x0, imm`；32位立即数：`lui + addi`组合 |
| `j label` | 无条件跳转到label | `jal x0, label` |
| `ret` | 函数返回 | `jalr x0, 0(x1)` |
| `nop` | 空操作，不执行任何有效动作 | `addi x0, x0, 0` |

## 十六、Java程序的执行机制
Java程序采用**跨平台的虚拟机架构**，与C语言的本地编译执行流程有本质区别，核心执行流程如下：
```
Java程序 → 编译器 → Class文件（Java字节码） → Java虚拟机（JVM） → 解释执行 / JIT即时编译 → 硬件执行
```

### 16.1 核心执行阶段
1. **编译阶段**：Java源码被编译为平台无关的**Java字节码（Bytecode）**，存储在Class文件中，字节码是JVM的标准指令集
2. **运行阶段**：JVM加载Class文件，采用两种方式执行字节码：
   - **解释执行**：逐条将字节码翻译为本地机器指令执行，启动快、执行效率低
   - **即时编译（JIT, Just In Time）**：识别高频执行的“热方法”，将其字节码一次性编译为本地机器码并缓存，后续直接执行编译后的机器码，大幅提升重复执行的性能

### 16.2 核心特性
- **可移植性**：一次编译，到处运行。字节码不依赖特定硬件ISA，只要目标平台有兼容的JVM即可运行，无需重新编译
- **性能权衡**：JIT编译弥补了解释执行的性能劣势，但需要额外的内存缓存编译后的代码，启动速度慢于本地编译的C程序
- **动态优化**：JVM可以在运行时基于程序执行特征进行动态优化，实现静态编译器难以达到的优化效果

## 十七、ISA设计的常见谬误（Fallacies）
### 17.1 谬误1：更强大的指令 → 更高的性能
- 错误认知：复杂的强大指令可以减少程序的总指令数，从而提升性能
- 事实真相：
  1. 复杂指令的硬件实现难度极高，会拖慢所有指令的执行周期，包括简单指令
  2. 现代编译器非常擅长用简单的RISC指令生成高效的机器码，复杂指令往往难以被编译器有效利用
  3. 手动编写汇编代码不一定能获得更高性能：现代编译器对现代处理器的流水线、缓存特性的优化能力，远超大多数人工编写的汇编代码；更多的汇编代码行数也意味着更高的出错概率与更低的开发效率

### 17.2 谬误2：向后兼容性 = 指令集不会改变
- 错误认知：为了保证向后兼容，处理器的ISA必须保持不变，不能新增指令
- 事实真相：
  1. 向后兼容仅要求旧指令的行为保持不变，不限制新增指令
  2. 主流ISA（如x86）在发展过程中持续新增大量扩展指令，从1978年到2012年，x86的指令数量与复杂度持续增长
  3. 新增指令可以在兼容旧程序的同时，为新程序提供更强的功能与更高的性能
