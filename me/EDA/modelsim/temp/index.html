    Temp

    你这篇笔记目前已经把 vlib / vmap / vlog / vsim 的 CLI 流程讲出来了：vlib 创建库，vmap 映射库，vlog/vcom 编译，vsim 加载并运行仿真。接下来讲 modelsim.ini 非常自然，因为它正好解释了 vmap 的映射结果到底保存在哪里，以及脚本化仿真为什么要管理这个文件。你的笔记中已经把这些命令的角色拆开了，尤其是 vmap 与库映射的关系，后续可以把 modelsim.ini 作为 “ModelSim 的仿真环境配置文件” 接在 CLI 章节后面。(smlcoke.com)

    1. modelsim.ini 是什么？¶

    modelsim.ini 是 ModelSim/Questa 在启动、编译、仿真时读取的 初始化配置文件。官方手册明确说，ModelSim 启动时会访问 modelsim.ini，其中包含初始工具设置；它还会根据一定优先级寻找应该使用哪一个 modelsim.ini。(ww1.microchip.com)

    对初学者来说，可以这样理解：

    modelsim.ini 就是 ModelSim 的“工程环境配置表”。
    它告诉 vlog/vcom/vsim/vmap：有哪些逻辑库、逻辑库对应哪个物理目录、编译器默认行为是什么、仿真器默认行为是什么、波形文件默认怎么保存等。

    它不是 Verilog/VHDL 源码，也不是编译后的库本身，而是 工具配置文件。真正的编译产物通常放在 work/ 或其他通过 vlib 创建出来的库目录中；modelsim.ini 负责记录这些库的名字和路径关系。

    2. 为什么它和 vmap 关系最大？¶

    ModelSim 的库有两个层次：

    概念 含义 例子
    逻辑库名 HDL 代码/仿真命令中使用的库名 work, unisim, altera_mf
    物理目录 磁盘上真实存放编译产物的文件夹 ./work, ./sim_lib/unisim

    vlib 负责创建物理库目录，例如：

    vlib work

    官方文档说明，创建库后目录中会包含 _info 文件，用来标识这是一个 ModelSim 设计库，而不是普通文件夹；因此不要直接用系统命令手动 mkdir 来代替 vlib。(ww1.microchip.com)

    vmap 则负责把逻辑库名映射到物理目录，例如：

    vmap work work
vmap unisim ./sim_lib/unisim

    官方文档明确说明，vmap <logical_name> <directory_pathname> 会把映射关系添加到 modelsim.ini 的 library section 中；也可以手动在 [Library] 下面写入 <logical_name> = <directory_pathname>。(ww1.microchip.com)

    也就是说：

    vmap work work

    本质上是在 modelsim.ini 里写入类似：

    [Library]
work = work

    所以 modelsim.ini 最核心、最常见的作用就是：

    保存 “逻辑库名 → 物理库目录” 的映射关系。

    这也是 Python 脚本集成 ModelSim 时必须关心它的原因。因为 Python 通过 subprocess 调用 vlib/vmap/vlog/vsim 时，这些命令需要知道当前使用的是哪个 modelsim.ini，否则可能出现 work 找不到、第三方库找不到、库映射被写到错误位置等问题。

    3. modelsim.ini 的基本结构¶

    官方手册列出了 modelsim.ini 的主要 section：[library] 存放库路径，[vcom] 控制 VHDL 编译，[vlog] 控制 Verilog 编译，[vsim] 控制仿真器，[DefineOptionset] 可定义常用命令行参数组合，[msg_system] 控制消息严重级别，[utils] 控制工具环境中的实用功能。(ww1.microchip.com)

    一个简化版可以长这样：

    [Library]
work = work
unisim = C:/Xilinx/Vivado/2023.2/data/simmodels/modelsim/unisim
altera_mf = D:/intelFPGA_lite/20.1/modelsim_ase/altera/verilog/altera_mf

[vlog]
; 控制 Verilog/SystemVerilog 编译器默认行为
; 例如是否多文件作为一个 compilation unit

[vsim]
; 控制仿真器默认行为
; 例如 WLF 波形文件名、WLF 是否压缩、默认 radix 等

[DefineOptionset]
; 定义常用参数组
; 例如 DEBUG = -voptargs=+acc

    其中最常见的是 [Library]，尤其是在课程设计、FPGA IP 仿真、Python 自动化仿真中。

    4. ModelSim 到底会使用哪一个 modelsim.ini？¶

    这是最容易踩坑的地方。ModelSim 不一定只读安装目录下的 modelsim.ini。官方手册给出的查找优先级大致是：

    1. 如果命令行使用了 -modelsimini <ini_filepath>，优先使用这个文件；
    2. 否则使用环境变量 MODELSIM 指定的 modelsim.ini；
    3. 否则尝试工作目录相关路径，例如 ./modelsim.ini；
    4. 再尝试安装目录下的 modelsim.ini；
    5. 如果都找不到，路径会退回到 ./modelsim.ini。(ww1.microchip.com)

    因此，下面几种情况可能会读到不同的 modelsim.ini：

    # 情况 1：当前目录有 modelsim.ini
vsim -c work.tb_top

# 情况 2：环境变量 MODELSIM 指向某个 ini
$env:MODELSIM = "D:/proj/sim/modelsim.ini"
vsim -c work.tb_top

# 情况 3：命令行强制指定 ini
vsim -modelsimini D:/proj/sim/modelsim.ini -c work.tb_top
vlog -modelsimini D:/proj/sim/modelsim.ini -work work rtl/top.sv

    官方文档还说明，可以用 ModelSim 的 where 命令查看当前到底使用了哪个 modelsim.ini。(ww1.microchip.com)

    5. 为什么不建议直接改安装目录下的 modelsim.ini？¶

    安装目录下通常会有一个全局 modelsim.ini，但官方文档说明，刚安装时这个文件通常是 read-only；如果要改，需要取消只读属性。(ww1.microchip.com)

    但实际工程中更推荐：

    不要污染安装目录下的全局 modelsim.ini，而是在每个工程目录下维护一份本地 modelsim.ini。

    原因有三个：

    第一，可复现。你的工程脚本、库映射、仿真选项都跟工程放在一起，换电脑或换路径时更容易重建。

    第二，避免权限问题。Windows 下安装目录可能需要管理员权限，vmap 写入失败或写到意料之外的位置。

    第三，避免不同工程互相污染。比如一个工程映射了 Xilinx 库，另一个工程映射了 Intel/Altera 库，如果都写进全局 modelsim.ini，后面会越来越乱。

    VHDLwhiz 的 ModelSim 命令教程也建议使用项目本地副本：可以用 vmap -c 从系统模板复制一份本地 modelsim.ini，再设置 MODELSIM=<project_dir>/modelsim.ini，之后 vmap 等依赖 ini 的命令就会使用这份本地文件。(VHDLwhiz)

    6. 推荐的工程目录组织方式¶

    你后面如果要写 Python 集成 ModelSim，我建议采用这种目录结构：

    my_project/
├─ rtl/
│  ├─ alu.sv
│  └─ regfile.sv
├─ tb/
│  └─ tb_top.sv
├─ sim/
│  ├─ modelsim.ini
│  ├─ work/
│  ├─ run.do
│  └─ wave.do
└─ scripts/
   └─ run_modelsim.py

    其中：

    sim/modelsim.ini

    只服务于当前工程。

    典型初始化流程：

    cd my_project/sim

# 1. 创建本地工作库
vlib work

# 2. 把逻辑库 work 映射到物理目录 work
vmap work work

# 3. 编译 Verilog/SystemVerilog
vlog -sv -work work ../rtl/alu.sv ../rtl/regfile.sv ../tb/tb_top.sv

# 4. 启动仿真
vsim -c work.tb_top -do "run -all; quit -f"

    执行完 vmap work work 后，本地 modelsim.ini 的 [Library] 里应该有：

    [Library]
work = work

    如果有第三方库，比如 Xilinx/Intel 仿真库，则可以继续映射：

    vmap unisim D:/xilinx_sim_lib/unisim
vmap altera_mf D:/intelFPGA_lite/20.1/modelsim_ase/altera/verilog/altera_mf

    对应 modelsim.ini 中会出现：

    [Library]
work = work
unisim = D:/xilinx_sim_lib/unisim
altera_mf = D:/intelFPGA_lite/20.1/modelsim_ase/altera/verilog/altera_mf

    7. 在 Python 集成 ModelSim 时应该怎么处理？¶

    Python 自动化时，最关键的是：每一次调用 vmap/vlog/vsim 都应该明确使用同一份 modelsim.ini。

    推荐做法有两种。

    做法 A：设置 MODELSIM 环境变量¶

    import os
import subprocess
from pathlib import Path

project_root = Path(__file__).resolve().parents[1]
sim_dir = project_root / "sim"
modelsim_ini = sim_dir / "modelsim.ini"

env = os.environ.copy()
env["MODELSIM"] = str(modelsim_ini)

def run(cmd):
    print(">", " ".join(map(str, cmd)))
    subprocess.run(cmd, cwd=sim_dir, env=env, check=True)

run(["vlib", "work"])
run(["vmap", "work", "work"])
run(["vlog", "-sv", "-work", "work", "../rtl/alu.sv", "../tb/tb_top.sv"])
run(["vsim", "-c", "work.tb_top", "-do", "run -all; quit -f"])

    这个方式的逻辑是：让所有 ModelSim 命令通过 MODELSIM 找到当前工程自己的 modelsim.ini。官方初始化顺序中，MODELSIM 环境变量的优先级高于当前目录和安装目录，因此适合自动化脚本。(ww1.microchip.com)

    做法 B：每条命令都显式加 -modelsimini¶

    run(["vmap", "-modelsimini", str(modelsim_ini), "work", "work"])
run(["vlog", "-modelsimini", str(modelsim_ini), "-sv", "-work", "work", "../rtl/alu.sv"])
run(["vsim", "-modelsimini", str(modelsim_ini), "-c", "work.tb_top", "-do", "run -all; quit -f"])

    官方文档说明，-modelsimini <ini_filepath> 可以让 vsim/vcom/vlog/vdel/vdir/vmake 等命令使用替代的初始化文件，并且这个选项会覆盖 MODELSIM 环境变量指定的文件。(ww1.microchip.com)

    我更推荐 Python 后端使用 做法 B，因为它最显式、最不容易受用户环境变量污染。尤其当你以后写成通用脚本、课程项目模板、CI/regression runner 时，显式传入 -modelsimini 更稳。

    8. modelsim.ini 除了库映射，还能控制什么？¶

    虽然初学阶段最常接触的是 [Library]，但 modelsim.ini 还可以控制很多默认行为。

    比如 [vlog] 中的 MultiFileCompilationUnit 可以控制 Verilog 文件是分开作为独立 compilation unit，还是合并成一个 multi-file compilation unit；官方文档也说明它可以被命令行参数 vlog -mfcu 或 vlog -sfcu 覆盖。(ww1.microchip.com)

    比如 [vsim] 中的 WLF 相关变量可以控制波形数据库文件：WLFFilename 指定默认 WLF 文件名，默认是 vsim.wlf；WLFDeleteOnQuit 控制仿真结束时是否删除 WLF；WLFFileLock 控制是否防止覆盖 WLF 文件。(ww1.microchip.com)

    因此，modelsim.ini 可以看作三类配置的集合：

    类型 常见 section 作用
    库映射 [Library] 让 work/unisim/altera_mf 等逻辑库名指向磁盘目录
    编译器默认选项 [vlog], [vcom] 控制 Verilog/VHDL 编译默认行为
    仿真器默认选项 [vsim] 控制仿真运行、波形 WLF、默认显示等行为

    不过对你的笔记来说，不建议一上来把所有变量铺开讲。重点应该放在：

    1. modelsim.ini 与 vmap 的关系；
    2. 本地 modelsim.ini 与全局 modelsim.ini 的区别；
    3. Python 自动化时如何固定使用某一个 modelsim.ini；
    4. 第三方 IP/厂商库仿真为什么经常要改 [Library]。

    9. 建议你放进笔记的版本¶

    下面这段可以直接改写进你的文章：

    IV. ModelSim 的环境配置文件：modelsim.ini¶

    在前面的 CLI 流程中，我们已经使用了 vlib, vmap, vlog, vsim：

    vlib work
vmap work work
vlog -sv -work work ../rtl/top.sv ../tb/tb_top.sv
vsim -c work.tb_top -do "run -all; quit -f"

    其中最容易被忽略的就是 vmap。vmap work work 并不是单纯在当前终端里临时登记一下库路径，而是会把 “逻辑库名 → 物理目录” 的映射关系写入 modelsim.ini 文件。

    也就是说，下面这条命令：

    vmap work work

    本质上会在 modelsim.ini 中生成类似这样的内容：

    [Library]
work = work

    因此，modelsim.ini 可以理解为 ModelSim/Questa 的“仿真环境配置表”。它告诉工具：

    • 有哪些逻辑库；
    • 每个逻辑库对应磁盘上的哪个目录；
    • vlog/vcom 编译器默认使用哪些选项；
    • vsim 仿真器默认使用哪些选项；
    • WLF 波形数据库、消息等级、默认显示格式等工具行为如何设置。

    其中最常见、最重要的是 [Library] section。比如：

    [Library]
work = work
unisim = D:/xilinx_sim_lib/unisim
altera_mf = D:/intelFPGA_lite/20.1/modelsim_ase/altera/verilog/altera_mf

    这表示：

    • HDL/仿真命令中的 work 逻辑库，对应当前目录下的 work 文件夹；
    • unisim 逻辑库，对应 Xilinx 预编译仿真库；
    • altera_mf 逻辑库，对应 Intel/Altera 预编译仿真库。

    需要注意的是，vlib 和 vmap 是两件不同的事情：

    vlib work

    表示创建一个 ModelSim 设计库目录；而：

    vmap work work

    表示把逻辑库名 work 映射到物理目录 work。

    所以，推荐的最小初始化流程是：

    vlib work
vmap work work
vlog -sv -work work ../rtl/*.sv ../tb/tb_top.sv
vsim -c work.tb_top -do "run -all; quit -f"

    在工程实践中，不建议直接修改 ModelSim 安装目录下的全局 modelsim.ini。更好的做法是每个工程维护自己的本地 modelsim.ini，例如：

    project/
├─ rtl/
├─ tb/
├─ sim/
│  ├─ modelsim.ini
│  └─ work/
└─ scripts/
   └─ run_modelsim.py

    这样做的好处是：

    • 不污染全局 ModelSim 环境；
    • 不需要管理员权限修改安装目录；
    • 不同项目之间的库映射互不影响；
    • Python 脚本、CI、regression test 更容易复现。

    在 Python 自动化仿真中，尤其应该显式指定当前工程使用哪一个 modelsim.ini。可以通过环境变量：

    env = os.environ.copy()
env["MODELSIM"] = str(sim_dir / "modelsim.ini")
subprocess.run(["vlog", "-sv", "-work", "work", "../rtl/top.sv"], cwd=sim_dir, env=env)

    也可以在每条 ModelSim 命令中显式加入：

    subprocess.run([
    "vlog",
    "-modelsimini", str(sim_dir / "modelsim.ini"),
    "-sv",
    "-work", "work",
    "../rtl/top.sv"
], cwd=sim_dir)

    对于自动化脚本来说，后一种方式更稳，因为它不会受到用户系统环境变量的影响。这样可以保证 vmap, vlog, vsim 使用的是同一份工程本地 modelsim.ini，避免出现 “编译时能找到 work，仿真时找不到 work” 或 “第三方库映射丢失” 这类问题。

    10. 最适合你笔记的总结句¶

    你可以在这一节最后放一句：

    modelsim.ini 不是仿真源码，也不是编译产物，而是 ModelSim 的环境配置文件。
    它最核心的作用是保存逻辑库到物理目录的映射关系，因此 vmap 实际上就是在维护 modelsim.ini。在 Python 自动化调用 vmap/vlog/vsim 时，应该为每个工程使用独立的本地 modelsim.ini，并通过 MODELSIM 环境变量或 -modelsimini 参数显式指定它。
